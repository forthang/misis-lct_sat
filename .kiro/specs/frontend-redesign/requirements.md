# Requirements Document

## Introduction

Редизайн фронтенд-приложения аналитического дашборда Газпромбанка для анализа клиентских отзывов. Цель — создать современный, минималистичный и визуально привлекательный интерфейс уровня Awwwards, сохранив весь существующий функционал. Дизайн должен быть интуитивно понятным, с плавными анимациями и премиальным внешним видом.

## Glossary

- **Dashboard**: Главная панель управления с визуализацией данных
- **Clustering Page**: Страница кластеризации отзывов по темам
- **Heatmap**: Тепловая карта негативных отзывов
- **Sidebar**: Боковая навигационная панель
- **Topic Card**: Карточка выбора темы/кластера
- **Time Range Selector**: Компонент выбора временного диапазона
- **Chart**: Графический компонент визуализации данных
- **Mock Data**: Заглушки данных для демонстрации функционала
- **API Integration**: Взаимодействие с бэкенд-сервером

## Requirements

### Requirement 1: Общий визуальный стиль

**User Story:** Как пользователь, я хочу видеть современный минималистичный интерфейс, чтобы работа с дашбордом была приятной и эффективной.

#### Acceptance Criteria

1. THE Dashboard SHALL использовать цветовую палитру с основным цветом #0A0A0B (темный фон) и акцентными цветами для контраста
2. THE Dashboard SHALL применять шрифт Inter с различными весами (300-700) для типографики
3. THE Dashboard SHALL использовать плавные анимации с длительностью 200-400ms для всех интерактивных элементов
4. THE Dashboard SHALL применять систему отступов на основе 8px сетки
5. THE Dashboard SHALL использовать скругления углов 12-24px для карточек и компонентов

### Requirement 2: Боковая навигация (Sidebar)

**User Story:** Как пользователь, я хочу иметь удобную навигацию по разделам дашборда, чтобы быстро переключаться между функциями.

#### Acceptance Criteria

1. THE Sidebar SHALL отображать логотип и название приложения в верхней части
2. THE Sidebar SHALL содержать иконки для каждого пункта меню с подписями
3. WHEN пользователь наводит курсор на пункт меню THEN THE Sidebar SHALL показывать плавную анимацию подсветки
4. WHEN пользователь выбирает пункт меню THEN THE Sidebar SHALL выделять активный пункт акцентным цветом
5. THE Sidebar SHALL сворачиваться до иконок на мобильных устройствах с шириной экрана менее 768px

### Requirement 3: Страница кластеризации

**User Story:** Как аналитик, я хочу видеть распределение отзывов по темам и тональности, чтобы понимать общую картину обратной связи.

#### Acceptance Criteria

1. THE Clustering Page SHALL отображать карточки тем в виде горизонтального скроллируемого списка с эффектом glassmorphism
2. WHEN пользователь выбирает тему THEN THE Clustering Page SHALL обновлять графики с плавной анимацией перехода
3. THE Clustering Page SHALL отображать линейный график динамики отзывов с градиентной заливкой под линией
4. THE Clustering Page SHALL отображать круговую диаграмму распределения с интерактивными сегментами
5. WHEN данные загружаются THEN THE Clustering Page SHALL показывать скелетон-анимацию вместо пустого пространства

### Requirement 4: Выбор временного диапазона

**User Story:** Как аналитик, я хочу фильтровать данные по временным периодам, чтобы анализировать тренды за разные промежутки времени.

#### Acceptance Criteria

1. THE Time Range Selector SHALL отображать предустановленные периоды в виде pill-кнопок
2. WHEN пользователь выбирает период THEN THE Time Range Selector SHALL выделять активную кнопку с анимацией
3. THE Time Range Selector SHALL предоставлять возможность выбора произвольного диапазона дат через date picker
4. WHEN пользователь изменяет временной диапазон THEN THE Dashboard SHALL автоматически обновлять все графики

### Requirement 5: Карточки и графики

**User Story:** Как пользователь, я хочу видеть данные в красиво оформленных карточках, чтобы информация была легко воспринимаемой.

#### Acceptance Criteria

1. THE Card component SHALL иметь эффект glassmorphism с полупрозрачным фоном и размытием
2. THE Card component SHALL отбрасывать мягкую тень для создания глубины
3. WHEN пользователь наводит курсор на карточку THEN THE Card component SHALL показывать легкий эффект подъема
4. THE Chart components SHALL использовать градиентные цвета для визуализации данных
5. THE Chart components SHALL показывать интерактивные тултипы при наведении на точки данных

### Requirement 6: Тепловая карта

**User Story:** Как аналитик, я хочу видеть концентрацию негативных отзывов на тепловой карте, чтобы выявлять проблемные области.

#### Acceptance Criteria

1. THE Heatmap SHALL отображать данные в виде цветовой матрицы с градиентом от светлого к темному
2. THE Heatmap SHALL показывать подписи осей с названиями тем и периодов
3. WHEN пользователь наводит курсор на ячейку THEN THE Heatmap SHALL показывать тултип с детальной информацией
4. THE Heatmap SHALL использовать цветовую шкалу, соответствующую общему дизайну дашборда

### Requirement 7: Страница рекомендаций ИИ

**User Story:** Как аналитик, я хочу получать рекомендации от ИИ по улучшению продуктов, чтобы принимать обоснованные решения.

#### Acceptance Criteria

1. THE Recommendations Page SHALL отображать выпадающий список для выбора темы анализа
2. THE Recommendations Page SHALL показывать кнопку генерации рекомендаций с индикатором загрузки
3. WHEN рекомендации загружаются THEN THE Recommendations Page SHALL показывать анимацию печатающегося текста
4. THE Recommendations Page SHALL форматировать markdown-текст рекомендаций с правильной типографикой

### Requirement 8: Страница сравнения с конкурентами

**User Story:** Как аналитик, я хочу сравнивать показатели банка с конкурентами, чтобы понимать позицию на рынке.

#### Acceptance Criteria

1. THE Competitor Page SHALL отображать чекбоксы для выбора конкурентов
2. THE Competitor Page SHALL показывать горизонтальную столбчатую диаграмму NPS
3. THE Competitor Page SHALL выделять показатели Газпромбанка акцентным цветом
4. WHEN данные недоступны THEN THE Competitor Page SHALL отображать mock-данные с соответствующим уведомлением

### Requirement 9: Страница генерации отчетов

**User Story:** Как аналитик, я хочу генерировать отчеты в различных форматах, чтобы делиться результатами анализа.

#### Acceptance Criteria

1. THE Reports Page SHALL предоставлять пошаговый интерфейс выбора параметров отчета
2. THE Reports Page SHALL отображать чекбоксы для выбора тем в виде сетки
3. THE Reports Page SHALL предоставлять выбор формата (Excel, PDF) через radio-кнопки
4. WHEN отчет генерируется THEN THE Reports Page SHALL показывать прогресс-индикатор

### Requirement 10: Страница оповещений

**User Story:** Как аналитик, я хочу видеть оповещения о резких изменениях, чтобы быстро реагировать на проблемы.

#### Acceptance Criteria

1. THE Alerts Page SHALL отображать карточки оповещений с цветовой индикацией критичности
2. THE Alerts Page SHALL показывать название темы, сообщение и процент изменения
3. WHEN оповещений нет THEN THE Alerts Page SHALL показывать сообщение об отсутствии активных оповещений

### Requirement 11: Страница тестирования

**User Story:** Как разработчик, я хочу загружать JSON-файлы для тестирования модели, чтобы проверять качество классификации.

#### Acceptance Criteria

1. THE Testing Page SHALL предоставлять drag-and-drop зону для загрузки файлов
2. THE Testing Page SHALL показывать индикатор загрузки во время обработки файла
3. THE Testing Page SHALL отображать кнопку скачивания результатов после обработки
4. WHEN файл загружен THEN THE Testing Page SHALL показывать уведомление об успешной загрузке

### Requirement 12: Адаптивность и мобильная версия

**User Story:** Как пользователь мобильного устройства, я хочу комфортно работать с дашбордом, чтобы анализировать данные в любом месте.

#### Acceptance Criteria

1. THE Dashboard SHALL адаптировать layout для экранов шириной менее 1200px, 768px и 480px
2. THE Dashboard SHALL использовать hamburger-меню для навигации на мобильных устройствах
3. THE Dashboard SHALL масштабировать графики для корректного отображения на малых экранах
4. THE Dashboard SHALL обеспечивать touch-friendly размеры интерактивных элементов (минимум 44x44px)

### Requirement 13: Интеграция с API и Mock-данные

**User Story:** Как пользователь, я хочу видеть реальные данные или демо-данные, чтобы понимать работу системы.

#### Acceptance Criteria

1. THE Dashboard SHALL пытаться загрузить данные из API при инициализации
2. WHEN API недоступен THEN THE Dashboard SHALL использовать mock-данные для демонстрации
3. THE Dashboard SHALL показывать индикатор источника данных (API/Mock) в интерфейсе
4. THE Dashboard SHALL корректно обрабатывать ошибки API и показывать понятные сообщения

### Requirement 14: Анимации и микровзаимодействия

**User Story:** Как пользователь, я хочу видеть плавные анимации, чтобы интерфейс ощущался отзывчивым и современным.

#### Acceptance Criteria

1. THE Dashboard SHALL применять ease-out анимации для появления элементов
2. THE Dashboard SHALL использовать spring-анимации для интерактивных элементов
3. THE Dashboard SHALL показывать skeleton-загрузчики во время ожидания данных
4. THE Dashboard SHALL применять stagger-анимации для списков элементов
